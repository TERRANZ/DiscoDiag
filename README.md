# DiscoDiag

**DiscoDiag** — кроссплатформенное десктопное приложение для диагностики автомобиля **Land Rover Range Rover L322 (5.0 supercharged, 2010+)** через OBD2 интерфейс. Программа подключается к ЭБУ автомобиля по Bluetooth (RFCOMM) и отображает в реальном времени данные с датчиков температуры, давления и уровня жидкостей в виде приборной панели.

## Возможности

- **Подключение по Bluetooth** — сканирование доступных Bluetooth-адаптеров и сервисов с последующим установлением RFCOMM-соединения с ЭБУ автомобиля.
- **Инициализация OBD2** — автоматическая последовательность инициализации: сброс ЭБУ (`ATZ`), включение заголовков ответов (`ATH1`), выбор протокола (`ATSP6`).
- **Непрерывный мониторинг** — циклический опрос датчиков в фоновом потоке с отображением данных в реальном времени.
- **Приборная панель** — отображение показаний в виде LCD-индикаторов и поворотных циферблатов (QDial).
- **Модульная архитектура команд** — паттерн Command позволяет легко добавлять новые диагностические команды.
- **Логирование** — вывод отладочной информации в консоль и файл `debug.log`.
- **Поддержка русского языка** — интерфейс приложения полностью переведён на русский язык (QTranslate).

## Архитектура

Приложение построено по слоистой событийно-управляемой архитектуре с четким разделением ответственности.

### Слои приложения

```
┌─────────────────────────────────────────────────────┐
│                    UI Layer                         │  (src/ui/)
│        MainWindow, RemoteSelector, *.ui             │
├─────────────────────────────────────────────────────┤
│                Bluetooth Layer                      │  (src/bt/)
│                    BtBackend                        │
├─────────────────────────────────────────────────────┤
│                  OBD Layer                          │  (src/obd/)
│         ObdService, ObdThread, ObdParser            │
├─────────────────────────────────────────────────────┤
│               Command Layer                         │  (src/command/)
│         AbstractCommand + implementations           │
└─────────────────────────────────────────────────────┘
```

### Поток данных

```
Пользователь нажимает "Запуск..."
              │
              ▼
  MainWindow::on_action_start_triggered()
              │
              ▼
    Создание ObdThread (фоновый поток)
              │
              ▼
    ObdService::startService()
              │
              ├──→ BtBackend::listAdapters()  (проверка Bluetooth)
              │
              ▼
    BtBackend::startClient()
              │  QBluetoothSocket → 10:21:3E:4A:3E:41 (RFCOMM)
              ▼
    ObdService::connected()  [Состояние: CONNECTED]
              │
              ▼
    Последовательность инициализации:
    ┌──────────────────────────────────────────────┐
    │ 1. ATZ  (ObdResetFixCommandImpl)   NC→DEV_SELECTED→CONNECTED |
    │ 2. ATH1 (DisplayHeaderCommandImpl) CONNECTED→RESETTED→DEV_CONFIG   |
    │ 3. ATSP6 (SelectProtocolObdCommandImpl) DEV_CONFIG→RESETTED→INWORK  │
    └──────────────────────────────────────────────┘
              │
              ▼
    ObdService::doObdLoop()  [Состояние: INWORK]
              │
              │  Циклический опрос (round-robin):
              │  22F405 → 22F446 → 22F40F → 2203F3 → 221E69 → 22D11A → (повтор)
              │
              ▼
    Обработка ответа:
    - ObdParser извлекает данные из hex-строки
    - Command::calculate() преобразует байты в значение
    - ObdService формирует ObdResult
    - emit updateUI(result)
              │
              ▼
    MainWindow::update_ui(result)
              │
              ├──→ teDebug: append (отладочный вывод)
              ├──→ numTempCooland->display()  +  dTempCoolant->setValue()
              ├──→ numTempAirAmb->display()   +  dTempAirAmb->setValue()
              ├──→ numTempAirInt->display()   +  dTempAirInt->setValue()
              ├──→ numTempOil->display()      +  dTempOil->setValue()
              └──→ numTempGb->display()       +  dTempGb->setValue()
```

    │    NC    │  startService()

         │                      │                    │        ┌─────────────┐  ATH1→OK

## Диагностические команды

### Паттерн Command

Все OBD-команды наследуются от абстрактного класса `AbstractCommand`:

```cpp
class AbstractCommand {
    QString  m_cmdId;      // OBD2-идентификатор команды (например "22F405")
    QString  m_cmdName;    // Человеко-читаемое имя команды
    QString  m_rawData;    // Сырые данные ответа

    virtual int calculate(const QString &value);
    // По умолчанию возвращает -1; переопределяется в подклассах
    // для преобразования сырых hex-данных в осмысленное значение
};
```

### Общие OBD2 AT-команды

| Класс | Команда | Назначение |
|---|---|---|
| `ObdResetFixCommandImpl` | `ATZ` | Сброс ЭБУ / OBD-адаптера |
| `DisplayHeaderCommandImpl` | `ATH1` | Включение заголовков в ответах (ЭБУ будет prepend'ить ID команды) |
| `SelectProtocolObdCommandImpl` | `ATSP6` | Выбор протокола 6 (KWP2000 fast initialization) |
| `ExtParamsCommandImpl` | *(пусто)* | Расширенные параметры (заглушка, не реализована) |

### Команды датчиков автомобиля (LR322)

**Температура** — все используют одинаковую формулу `byte6 - 40` (стандарт OBD2 для температуры в °C):

| Класс | Команда | Датчик |
|---|---|---|
| `CoolantTempCommandImpl` | `22F405` | Охлаждающая жидкость |
| `OilTempCommandImpl` | `2203F3` | Масло двигателя |
| `GbTempCommandImpl` | `221E69` | Коробка передач |
| `TCTempCommandImpl` | `22D11C` | Раздаточная коробка |
| `RDTempCommandImpl` | `221E8A` | Задний дифференциал |
| `ComprTempCommandImpl` | `22D11A` | Компрессор |
| `IntakeAirTempCommandImpl` | `22F40F` | Воздух на впуске |
| `AmbientAirTempCommandImpl` | `22F446` | Наружный воздух |

**Давление и уровень:**

| Класс | Команда | Датчик | Формула |
|---|---|---|---|
| `FuelPressCommandImpl` | `22F423` | Давление топлива | `(byte6 × 256 + byte7) / 100` |
| `BrakePressCommandImpl` | `222B0D` | Тормозная система | `byte6 × 8533.3337 + byte7 × 33.33` |
| `FuelLevelCommandImpl` | `22F42F` | Уровень топлива | `(byte6 × 100) / 255` (литры) |
| `OilLevelCommandImpl` | `2203E6` | Уровень масла | *(stub — не реализована)* |
| `OilVolumeCommandImpl` | `2203F2` | Объём масла | *(stub — не реализована)* |

## Протокол общения с ЭБУ

### Подключение

| Параметр | Значение |
|---|---|
| **Протокол** | RFCOMM (Serial Port Service Class UUID) |
| **Адрес Bluetooth** | `10:21:3E:4A:3E:41` (хардкод) |
| **Терминатор команды** | `\r` (возврат каретки) |
| **Терминатор ответа** | `>` (знак больше) |

### Формат ответа

После включения `ATH1` ЭБУ prepend'ит ID команды к ответам. Пример ответа на запрос `22F405`:

```
22F405 7E8 04 62 F4 05 5C
│    │   │   │  │  │  │
│    │   │   │  │  │  └── Контрольная сумма (checksum)
│    │   │   │  │  └───── Байт данных (позиция 6) → raw sensor value
│    │   │   │  └──────── Байт данных (позиция 5)
│    │   │   └─────────── SID 0x62 (подтверждающий ответ)
│    │   └─────────────── Количество данных байт
│    └─────────────────── Идентификатор ЭБУ (ECU identifier)
└──────────────────────── Эхо отправленной команды
```

### Парсинг ответов

Класс `ObdParser` предоставляет статические методы:

| Метод | Описание |
|---|---|
| `A(const QString &data)` | Возвращает целое значение из позиции 6 (сырые данные датчика) |
| `B(const QString &data)` | Возвращает целое значение из позиции 7 |
| `getIntAtPos(data, pos)` | Общее извлечение hex-значения по позиции |
| `extractCommandId(data)` | Извлечение ID команды из первого токена |

## Установка и сборка

### Требования

- **C++ компилятор** с поддержкой C++11 (GCC 7+, Clang 5+, MSVC 2017+)
- **Qt 6** (модули: Core, Widgets, Bluetooth)
  - Автоматически деградирует до Qt 5, если Qt 6 не обнаружен
- **CMake 3.10+**
- **BlueZ** (на Linux) — стек Bluetooth

### Локальная сборка

```bash
# Клонирование репозитория
git clone <repository-url>
cd DiscoDiag

# Создание директории сборки
mkdir build && cd build

# Конфигурация и сборка
cmake .. -DCMAKE_BUILD_TYPE=Release
make -j$(nproc)
```

### Конфигурации сборки

| Конфигурация | Описание |
|---|---|
| `Debug` | Для отладки, с дебаг-символами |
| `Release` | Оптимизированная версия |
| `MinSizeRel` | Минимальный размер бинарного файла |

### Результат сборки

Бинарный файл: `build/DiscoDiag`

## Использование

### Запуск

```bash
./build/DiscoDiag
```

При запуске создаётся файл `debug.log` в рабочей директории, в который записывается отладочная информация.

### Интерфейс

Меню приложения:

| Меню | Действия |
|---|---|
| **Диагностика** | `Запуск...` — начать опрос датчиков, `Выход` — закрыть приложение |
| **Настройка** | `Настройки приложения...` — открыть диалог выбора Bluetooth-устройства |
| **Помощь** | `О программе...` — показать информацию о приложении |

### Приборная панель

Приборная панель разделена на два ряда:

**Первый ряд — основные температуры:**
- Температура охлаждающей жидкости
- Температура наружного воздуха
- Температура на впуске
- Температура масла
- Температура коробки передач

**Второй ряд — дополнительные параметры:**
- Температура заднего дифференциала (ЗД)
- Температура раздаточной коробки
- Температура компрессора
- Уровень масла
- Давление масла

Каждый параметр отображается в виде поворотного циферблата (QDial, диапазон: -120…120) и цифрового значения (QLCDNumber).

Внизу расположена текстовая область для отладки (`teDebug`), отображающая сырые OBD-ответы.

## CI/CD

### GitLab CI Pipeline

Проект использует GitLab CI для автоматической сборки.

```yaml
image: 192.168.3.1:5000/vterranz/discobuild

before_script:
  - export DOCKER_HOST=http://192.168.1.3:2375

build:
  stage: build
  script:
    - mkdir build && cd build && cmake .. && make -j
  artifacts:
    paths:
      - build/DiscoDiag
```

Docker-образ для сборки проекта находится в `builder/Dockerfile`. Он содержит:

## Примечания по разработке

### Именование файлов

| Тип | Расширение | Пример |
|---|---|---|
| Заголовочные файлы | `.h` | `BtBackend.h` |
| Исходные файлы | `.cpp` | `BtBackend.cpp` |
| Qt Designer | `.ui` | `mainwindow.ui` |
| Qt Translation | `.ts` | `DiscoDiag_ru_RU.ts` |

### Стиль кода (Qt Creator)

- Отступы: **4 пробела**
- Кодировка: **UTF-8**
- Ширина строки: **80 символов**

### Область текущего статуса

**Реализовано и активно в цикле опроса:**
- 6 температурных датчиков (охлаждающая жидкость, наружный воздух, впуск, масло, коробка, компрессор)

**Реализованы, но НЕ в цикле опроса:**
- Давление топлива (`22F423`)
- Давление тормозной системы (`222B0D`)
- Уровень топлива (`22F42F`)

**UI виджеты существуют, но данные не обновляются:**
- Температура раздатки, задний дифференциал, компрессор
- Давление масла, уровень/объём масла

**Заглушки (stub, без реализации `calculate()`):**
- Температура заднего дифференциала (`RDTempCommandImpl`)
- Уровень масла (`OilLevelCommandImpl`)
- Объём масла (`OilVolumeCommandImpl`)
- Расширенные параметры (`ExtParamsCommandImpl`)

### Известные особенности

1. **Bluetooth-адрес зашит жёстко** — `10:21:3E:4A:3E:41`. Диалог `RemoteSelector` позволяет сканировать устройства, но выбранное устройство не передаётся в `BtBackend`.

2. **Разрыв между UI и логикой** — `MainWindow::update_ui()` обновляет только 5 датчиков, хотя в UI-файле объявлено значительно больше виджетов.

3. **Пропущенные команды** — 5 из 13 определённых в `commandIds.h` команд не зарегистрированы в `ObdService` для циклического опроса.

4. **Потоки** — приложение использует `QThread` для выполнения `ObdService` в фоновом потоке. Сигналы/слоты Qt обеспечивают потокобезопасную коммуникацию между UI и OBD-слоем.
